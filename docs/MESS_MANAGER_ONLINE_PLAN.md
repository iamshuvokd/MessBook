## Role hierarchy & business model (decided 2026-07-15, second session)

```
MASTER ADMIN (the user only; Gmail(s) pinned in server env config)
  └─ manages buyers via a WEB DASHBOARD on the same droplet (master.<domain>):
     list buyers/messes, activate/deactivate, "Extend 30 days", basic stats
APP ADMIN (the buyer / mess owner)
  └─ pays the user directly (bKash etc., outside the app); Master extends
     `paid_until` manually. Expired mess → server flips it READ-ONLY
     (data safe, gently locked until renewal).
  └─ appoints SUB-ADMINS with CUSTOMISABLE permission flags:
       meals.manage · polls.create · polls.manage · expenses.manage ·
       money.manage (deposits/settlements/general close) · members.manage
     Named presets = one-tap bundles: Meal Admin (meals.manage+polls.*),
     Expense Admin (expenses.manage+money.manage), Poll Creator
     (polls.create), Member Admin (members.manage).
MEMBER — votes in polls, sees own balances/meals.
```

Schema impact: `group_members.role` ENUM('app_admin','sub_admin','member') + `permissions_json`; `groups.status` ENUM('active','expired','disabled') + `paid_until` DATE; `master_admins` allowlist via env. All permission checks server-side; the app mirrors them for UI gating (and keeps working offline where the local device owner is implicitly the App Admin).

**Core design rule (user, 2026-07-15): "fully customizable app" —** every mess-level behavior is an App-Admin-editable setting (meal separation, poll types, non-voter policy, categories, month start, etc.), and EVERY edit action anywhere in the app is gated by the role/permission system — edit controls are hidden from members whose role doesn't grant them. Nothing mess-specific is hardcoded beyond the App Admin's reach.

**Expenses specifically (user):** fully admin-customizable — custom categories (add/rename/icon/meal-flag), who may add/edit/delete expenses, deposits and settlements all controlled by the `expenses.manage` / `money.manage` permissions the App Admin grants.

**UI/UX rule (user): keep it SIMPLE.** Members see a clean read-and-vote app with zero admin clutter; admin controls appear only for the roles that hold them; permission assignment = tap a member → tap preset chips or simple on/off switches per duty. Plain language (bn+en), no settings mazes.

## Architecture

**Doctrine (user, re-confirmed 2026-07-15): the app is offline-based; it syncs with online when it gets data/connectivity.** Every feature must be fully usable with zero internet against the local Drift DB; the server is a synchronization + multi-device layer, never a dependency for core function. Any new feature that can't work offline (chat delivery, live votes from other phones) must queue locally and degrade gracefully.

```
Flutter app (Drift/SQLite, offline-first — unchanged)
        ⇅ HTTPS (JWT)
Node.js + Express API  ←  verifies Google ID tokens (google-auth-library)
        ⇅
MySQL 8 (source of truth for synced messes)
```

- The phone **never** talks to MySQL directly; only to the API.
- Every existing table already carries `updatedAt` (epoch ms) — designed for sync from day one.
- Offline behavior unchanged: all writes land in local Drift first; a sync service pushes/pulls when online. Last-write-wins by `updatedAt`; deletes are tombstones (`deleted` flag).
- Roles enforced **server-side**: `manager` | `meal_admin` | `member`, plus `can_create_poll` grant.

### API endpoints

Auth: `POST /auth/google {idToken}` → verify → upsert user → `{access, refresh, user}` · `POST /auth/refresh` · `POST /auth/logout` · `GET /me`
Groups: `POST /groups` (create + invite code) · `POST /groups/join {code, memberName}` · `GET /groups` · `PATCH /groups/:id/members/:mid/role` (manager only) · `DELETE /groups/:id/members/:mid`
Sync: `POST /groups/:id/sync/pull {sinceMs}` → changed rows per table · `POST /groups/:id/sync/push {changes}` → LWW upserts, returns per-row accept/conflict
Polls: `POST /groups/:id/polls` (role-checked) · `GET /groups/:id/polls?date=` · `POST /polls/:id/vote` (rejected after `close_at`) · close is evaluated server-side by timestamp (no cron needed).

## Meal slots & auto meal routine (user decision, 2026-07-15 — "customizable auto meal set up")

**Meal slots (per mess, App-Admin customizable):** new `meal_slots` table (id, group_id, name, weight, sort_order, active). Seeded per group as Breakfast (weight 0.5), Lunch (1), Dinner (1) — the admin can rename, add, remove slots and change weights (BD messes commonly count breakfast as a half meal; some count it full — hence customizable). A day's meal count = sum of taken slots' weights.

**Member auto-routine:** new `member_meal_routines` table (member_id, slot_id, weekday 0–6 or NULL=every day, enabled). Each member (or the Meal Admin for them) sets a standing routine — e.g. "Lunch + Dinner daily, no Breakfast, everything off on Friday". Plus **meal leave**: a date-range pause (member_id, from_date, to_date) for "going home 20–25th", which beats the routine.

**Auto-fill engine (offline, runs in appOpenTasksProvider):** each day, for members with a routine and no existing entry for that day, materialize the day's meal count from routine minus leave. Precedence, strongest first: **manual grid edit → poll vote → auto-routine → blank**. The existing `meals.count` stays the single authoritative number (engines/reports/grid untouched); a new nullable `meals.slotsJson` records which slots produced it, for display only.

**Poll integration:** poll types become `slots` (tick Breakfast/Lunch/Dinner — replaces plain on/off), `count`, `menu`. The non-voter policy gains a fourth, best-default option: **`routine`** — a non-voter simply gets their standing routine (alongside 'pending' | 'zero' | 'repeatYesterday'). All still admin-customizable per mess and per poll.

## Internal chat (user decision, 2026-07-15)

**Scope:** one group channel per mess (like a mess WhatsApp group) — not DMs. **Delivery:** real-time — messages arrive within ~1s, plus a push notification when the app is backgrounded/closed. This is the highest-infra feature in the plan; it rides on the same accounts+sync foundation, so it's built right after Step 6 (background sync), before online polls (which can reuse the same push channel for "poll closing soon" alerts).

Additions to the server (Node.js):
- `chat_messages` table (id, group_id, member_id, text, created_at, `client_nonce` for dedup on reconnect)
- **Socket.IO** (not raw WebSockets — reconnect/room support built in) layered onto the existing Express server: clients join a room per group (`group:<id>`) after authenticating the socket with the same JWT; `sendMessage`/`newMessage` events; history still fetched over plain REST (`GET /groups/:id/messages?before=`) for initial load/pagination.
- **Firebase Cloud Messaging (FCM)** for push when the socket isn't connected (app closed/killed): server holds each device's FCM token (`device_tokens` table), sends a push on new message to every mess member except the sender. FCM is part of the *same* Firebase/Google Cloud project as the Google Sign-In OAuth client from Step 4 — one setup covers both, just enable Cloud Messaging on that project.

Additions to the app (Flutter):
- `socket_io_client` package; a `ChatService` that connects/joins the group room when a synced mess is open, falls back to REST history when offline
- `firebase_messaging` package for receiving pushes and tapping through to the chat screen
- Simple chat UI: message list + input, sender name/avatar, "seen" not required for v1
- Chat is only available for **synced (online) messes** — offline-only messes don't show the chat tab, consistent with "chat needs everyone to have their own signed-in device"

✅ **Master Admin web dashboard done (2026-07-16)**: server — `GET /admin/groups` (every mess + owner email/name + active member count, master-admin-only via the existing `requireMasterAdmin`), `PATCH /admin/groups/:id` (set `status` and/or `paidUntil`; `loadGroupContext`'s existing lazy-expiry check does the rest — no cron, matching the established pattern). Dashboard itself is plain static HTML/CSS/vanilla JS at `public/admin/` (no build step, no framework) served by the *same* Express app at `/admin` — a deliberate simplification of the docker-task-list's original "own small container" note, since it's just static files reusing the same REST API; splitting it into its own container later is a one-line Dockerfile/compose change if ever wanted. Sign-in reuses Google Identity Services (the same Web OAuth client from Step 4) + the existing `/auth/google` endpoint; `/me`'s `isMasterAdmin` flag gates access (denied screen for any other Google account). Per-mess actions: editable paid-until date, "+30 days" shortcut, enable/disable toggle. **Caught and fixed a real bug during smoke-testing**: `adminRouter`'s auth middleware is registered with no path scope (matching every other router in this codebase), so mounting it *before* the static file middleware caused it to 401 the dashboard's own HTML/JS/CSS before they could load — fixed by serving `/admin` static assets first in the middleware stack, so only requests with no matching static file (i.e. the real API calls) fall through to the auth gate. Verified via `npm test` (2/2) and a live smoke-test (`curl` against a running instance): static assets 200, `/admin/groups` 401 unauthenticated, `/health` unaffected. **Needs from the user before it actually works live**: the Web OAuth client's "Authorized JavaScript origins" in Google Cloud Console needs the dashboard's real origin added (e.g. `http://localhost:3000` for local testing, the prod domain once deployed) — the Google Identity Services sign-in button silently fails on an unlisted origin.
